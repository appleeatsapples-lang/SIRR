## HIGH FINDINGS

### H1 — Methodology scholarship drift in 5 classical modules

**What I checked:** All 36 `keep_masters=...` call sites across
`Engine/modules/`. Cross-referenced `scholarship_fidelity` labels.

**What I found:** Five modules preserve Pythagorean master numbers
(11, 22, 33) on traditions that historically didn't have the
master-number concept:

- `greek_isopsephy.py:45` — Hellenistic isopsephy, 3rd–1st c BCE
- `coptic_isopsephy.py:23` — 1st–8th c Christian Egypt
- `armenian_gematria.py:20` — 5th c onwards
- `agrippan.py:22` — Cornelius Agrippa, 16th c Hermetic
- `trithemius.py:25` — Trithemius cipher, 15th–16th c

All 5 also lack `scholarship_fidelity` tags. They return
`certainty="COMPUTED_STRICT"` — i.e. claim the arithmetic is
authoritative — but the methodology itself is a modern synthesis.

This is the same drift class that the Grok R2 audit caught in
`mandaean_gematria`, `ethiopian_asmat`, and `hebrew_gematria`
(all now labeled `MODERN_SYNTHESIS`).

**Severity:** HIGH. Not a runtime bug — a scholarship integrity
issue that undermines the doctrine of "strict arithmetic on classical
systems, honestly labeled."

**Recommended action:**
- Option A (purist): drop `keep_masters=(11,22,33)` → `keep_masters=()`
  on these 5. Matches ethiopian/mandaean/hebrew treatment.
- Option B (product-design synthesis): keep and add
  `"scholarship_fidelity": "CLASSICAL_METHOD_MODERN_APPLICATION"` to
  each module's output dict.

Option B is lower disruption + more product-honest. One-hour PR.

### H2 — Raw `/reading/{order_id}` paths unauthenticated + guessable

**What I checked:** Access control on all 3 reading view variants.

**What I found:**
```
GET /reading/muhab-akif-23sep1996-9376          → 200 (no auth)
GET /reading/muhab-akif-23sep1996-9376/unified  → 200 (no auth)
GET /reading/muhab-akif-23sep1996-9376/merged   → 200 (no auth)
```

Order ID format is `{first}-{last}-{DDmonYYYY}-{4-digit-suffix}`
(generated from `uuid.uuid4().hex[:8]` per `order_store.py:30`, so
actually 8 hex chars = 4.3B possibilities — better than the 10K I
first estimated).

Still, the raw-path route exists as a "grandfathered" fallback
(`server.py:689` comment confirms). The §16.5 token architecture
exists specifically to keep order IDs off the wire. While raw paths
remain open, the token's privacy guarantee is half-effective.

**Severity:** HIGH. Not an immediate exploit (8-hex suffix isn't
trivially brute-forceable), but it undermines the §16 architecture
that the rest of the system was built around.

**Recommended action:** Deprecate `/reading/{order_id}` and variants.
Return 410 Gone with a message directing users to their signed link.
Tokens stay as the only way in. Tied to H3 below.

### H3 — PII leaking into Railway access logs via order_id in URL

**What I checked:** Recent Railway logs for the last 24h.

**What I found:**
```
INFO: ... GET /reading/muhab-akif-23sep1996-9376/merged HTTP/1.1" 200 OK
INFO: ... GET /reading/muhab-akif-23sep1996-9376/unified HTTP/1.1" 200 OK
INFO: ... GET /reading/muhab-akif-23sep1996-9376 HTTP/1.1" 200 OK
```

Uvicorn's default access log prints full paths. The order_id contains:
first name, last name, day/month/year of birth. Railway retains logs.
Any operator (including the auditor-Muhab role) can read these.

Procfile: `uvicorn server:app --host 0.0.0.0 --port ${PORT:-8000}`
No `--access-log false`. No custom log config.

Separately, 3 `print(... {e} ...)` statements in `server.py` and 3
`str(e)` assignments in `runner.py` could leak engine exception
messages containing user input to stderr / encrypted output:

- `server.py:1111` — `print(f"[unified_view] failed for order {order_id}: {e}", file=sys.stderr)`
- `server.py:1149` — `print(f"[merged_view] ... {e}", ...)`
- `server.py:1220` — `print(f"[legacy_reading] ... {e}", ...)`
- `runner.py:~1200` — `out["semantic_reading"] = {"error": str(e), ...}`
- `runner.py:~1210` — `out["psychological_mirror"] = {"error": str(e), ...}`
- `runner.py:~1220` — `out["psychological_profile"] = {"error": str(e), ...}`

Compare to the well-sanitized patterns already in the codebase:
`sanitize_exception(exc)` at admin/retention endpoints, and
`server.py:1240`'s `{type(enc_err).__name__}` pattern.

**Severity:** HIGH. Direct privacy leak in logs. Runner.py leaks are
encrypted at rest (lower risk) but decrypt into user-visible views.

**Recommended action:**
1. Add `--access-log false` to Procfile OR add a log-filter config that
   scrubs order_ids from access log lines.
2. Replace the 3 `print(... {e} ...)` calls in server.py with
   `print(f"... : {sanitize_exception(e)}", file=sys.stderr)`.
3. Replace the 3 `str(e)` in runner.py error handlers with
   `type(e).__name__` (just the class name — traceback not needed in
   user-facing output).
4. Pair with H2 deprecation: once `/reading/{order_id}` returns 410,
   the remaining `/r/{token}/...` paths don't expose any PII in URLs.

H2 and H3 should ship together as one PR — they solve the same
problem from different angles.

**What I checked:** File mtimes on `/data/readings/` vs commit history.

**What I found:** Muhab's Apr 20 `.html` and `_unified.html` still on

Legacy + unified views for existing orders are frozen on the version
of code that existed at checkout. Any customer who checked out before
PR #14 still sees 235 cards including 69 em-dash stubs.

**Severity:** MEDIUM. Doesn't affect new customers. Affects anyone
with a durable bookmark/link to an older reading.

**Recommended action:** Apply the same mtime-regen pattern (copy from
`server.py:870-888`) to `reading_page` and `reading_unified_page`
handlers. Or: one-time script to regenerate all existing HTML files.
One-time is simpler given there are currently 2 orders total on disk.

### M2 — No stuck-order detector

**What I checked:** `retention.py:sweep_tier2_expired` and
`purge_cycle`, order status transitions, background-thread lifecycle.

**What I found:** If the daemon thread generating a reading dies
mid-run (container kill, SIGKILL, OOM, C-extension panic), the order
stays at `status="processing"` forever. The retention sweep deletes
the order's files after `RETENTION_DAYS` (30) based on mtime, but
doesn't detect the stuck state itself or surface it to the customer.

Consequence: a customer who checked out during a crash sees the
loading spinner → eventually times out → has no path back to their
reading even though their order is still "processing" in the DB.

**Severity:** MEDIUM. Hasn't happened yet in prod (1 real order, no
crashes during its run). Will happen once traffic scales.

**Recommended action:** Add a stuck-order check to `purge_cycle`:
for orders in state `processing|pending|paid` older than 1 hour with
no `_output.json`, mark `status="failed"` with
`error="generation timed out"`. Surface a styled error page on the
success poller when status is `failed`.

## LOW FINDINGS

### L1 — CSP allows `'unsafe-inline'` for scripts and styles

Landing page has 1 `<script>` tag. Could be extracted to external file
+ replaced with SHA-256 hash in CSP. Styles have inline tags too.
Low priority — modern CSP convention is to tolerate inline where
alternatives are costly, but auditor-clean ideal is external + hash.

### L2 — Webhook returns 400 not 401 for bad signatures

`test_webhooks.py` tests assert 400 is correct, and 400 is semantically
defensible ("malformed request"), but 401 would be more precise. Not
worth changing unless you care about strict HTTP semantics. No security
impact.

### L3 — `sanitize_exc_now` legacy variant doesn't strip message

`sanitize.py:sanitize_exc_now` preserves the tail of `format_exc()`
which includes the exception message. Comment warns: "callers in
legacy code paths may rely on seeing the message tail." Potential
future drift if reused in new code. Recommend: audit all callers,
migrate to `sanitize_exception(exc)`, then delete the legacy helper.

### L4 — 405 errors on GET `/api/internal/purge`

Railway logs show occasional 405 on GET of a POST-only endpoint:
```
INFO: 100.64.0.X - GET /api/internal/purge HTTP/1.1 405 Method Not Allowed
```
Looks like a healthcheck or monitor hitting with wrong verb. Not
harmful but noisy. Either add a GET handler that returns a status,
or figure out what's probing it.
